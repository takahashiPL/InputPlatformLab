# T32: GDI debug overlay の D3D 側移行 — 候補比較（整理のみ・未実装）

前提: T30（render/resize/present の薄い整理）および T31（`WindowsRenderer_Init` / `Resize` / `Frame` / `Shutdown`）完了。現行 GDI overlay は正常。本書は **実装拡大せず**、T33 以降に渡す比較メモ。

---

## 1. 現状の描画順（WM_PAINT）

`MainApp.cpp` の `Win32_MainView_PaintFrame` より:

1. `BeginPaint`（`PAINTSTRUCT`・`HDC` 取得）
2. `WindowsRenderer_Frame` — D3D11 で RTV bind・viewport・clear・**Present**
3. `Win32DebugOverlay_Paint`（`WIN32_MAIN_D3D_CLEAR_ONLY_PAINT` 未定義時）— **GDI** でクライアント上にテキスト・スクロールクリップ
4. `EndPaint`

補足: **Present が先**で、その後 **同一 HWND の DC に GDI が重ね描き**される。`WM_ERASEBKGND` で既定の白塗りは抑止済み。

---

## 2. overlay を renderer 側へ寄せるときの責務境界（目標像）

| 層 | 責務 |
|----|------|
| **MainApp / WndProc** | `WM_PAINT` の入口、`BeginPaint` / `EndPaint`、入力・モード（T17 等）から渡すラベル文字列の供給（現状どおりでも可） |
| **WindowsRenderer** | スワップチェーン、RTV、クライアントサイズ、`ResizeBuffers`、**将来は overlay 用パスを同一デバイス上に載せる**、`Present` の最終タイミング |
| **Win32DebugOverlay（現状）** | GDI 専用。移行後は **削除または薄いラッパ** に縮退の可能性 |
| **InputCore 等** | 描画 API に非依存のまま維持 |

「renderer に寄せる」とは、**テキスト幾何・シェーダ・テクスチャ更新**を D3D デバイスコンテキスト側に集約し、**Present は 1 回**にまとめる構成を指す（下記「合成順」）。

---

## 3. clear / overlay / Present の順序

**現状（GDI 重ね）**

- clear → **Present** → GDI（DC）は「画面上の最終合成」が GDI になる。

**将来（D3D 内 overlay）**

- clear（バックバッファ）
- overlay 描画（同一 RTV または中間 RTV へ）
- **Present（1 回）**

スクロール付き長文は、**RTV 上でクリップ矩形相当**をシェーダ／ビューポート／シザーで再現するか、**中間テクスチャに一度描いてから貼る**かを別途決める。

---

## 4. resize 時に誰が何を持つべきか

| 主体 | 持つもの |
|------|----------|
| **WindowsRenderer** | `clientWidth` / `clientHeight`、`ResizeBuffers`、RTV 再作成、viewport。将来は **overlay 用バッファ／フォント解像度 DPI** もここで無効化・再作成が自然 |
| **MainApp** | `WM_SIZE` → `GetClientRect` → `WindowsRenderer_Resize` の呼び出しのみ（T31 どおり）。**オーバーレイの論理スクロール量**（`s_paintScrollY` 等）は今どおり MainApp / overlay モジュールが保持してよいが、**ピクセルバッファの寿命**は renderer 側に寄せると一貫 |
| **GDI 暫定** | リサイズ後 `InvalidateRect` で `WM_PAINT` が走り、GDI が新サイズで再描画 — 既存の流れを壊さない |

---

## 5. GDI を残す暫定案と、完全移行案の違い

| | **GDI 暫定（現状）** | **完全 D3D 移行** |
|--|----------------------|---------------------|
| Present | D3D が先に Present、GDI が上書き | **Present は最後に 1 回** |
| スクロール | `ScrollWindowEx` / クリップ領域が GDI 前提 | テクスチャオフセット or カスタムクリップ |
| DPI / フォント | `LOGFONT` / GDI フォント | DirectWrite またはビットマップフォント |
| デバッグ容易さ | 高い | 初期は低い（パイプライン増） |

## 6. 候補比較（3＋補助）

### 6.1 ビットマップフォント／フォントテクスチャ＋スプライト

| 観点 | 内容 |
|------|------|
| **良い点** | 実装が単純、描画コスト予測しやすい、既存ゲーム手法と親和 |
| **悪い点** | 日本語・可変幅・サイズ変更でアトラス肥大、デバッグ UI の「とりあえず長文」には向きにくい |
| **本リポへの適合** | 中 — プロトタイプ向き、本デバッグオーバーレイの要件（多言語・長文）とはややずれる |
| **先に必要な準備** | アトラス更新戦略、UV 管理、または ASCII 限定に割り切る |

### 6.2 DirectWrite（＋必要なら Direct2D / D2D-D3D11 相互運用）

| 観点 | 内容 |
|------|------|
| **良い点** | 日本語・レイアウト・ClearType に強い、Windows ネイティブ、デバッグ文字列に向く |
| **悪い点** | D2D/DWrite ファクトリ・デバイスコンテキスト・D3D との同期の学習コスト、初期コード量 |
| **本リポへの適合** | **高** — 現行が「デバッグ用長文テキスト」中心 |
| **先に必要な準備** | D3D11 デバイスと共有する D2D/DWrite の作成手順の固定、リサイズ時の再レイアウト |

### 6.3 中間テクスチャ／共有サーフェス（GDI オフスクリーン → GPU 合成）

| 観点 | 内容 |
|------|------|
| **良い点** | 既存 GDI 描画ロジックを一部再利用しやすい、段階的移行の橋渡しになりうる |
| **悪い点** | コピー回数・帯域・同期が増えやすい、最終的に二重メンテになりがち |
| **本リポへの適合** | 中 — 短期の「見た目だけ D3D 内」には使えるが、長期の第一選択ではない |
| **先に必要な準備** | DIB セクション／`UpdateSubresource` 経路の設計、リサイズ時のテクスチャ再割り当て |

### 6.4 補助: DXGI 共有テクスチャ・別プロセス合成

| 観点 | 内容 |
|------|------|
| **良い点** | 特殊なマルチプロセス UI に有効 |
| **悪い点** | 本プロジェクト規模では過剰 |
| **適合** | 低 |

## 7. 見送り案と理由

| 案 | 理由 |
|----|------|
| **完全カスタム SDF テキスト** | 品質は良いが、ラボ目的（入力・表示基盤の学習）に対し実装コストが大きすぎる |
| **ImGui 等の即時導入** | 依存・スタイル・入力フォーカスと既存メニュー統合の整理が別タスク化しやすい。**除外ではないが T32 の「薄い整理」の範囲外** |
| **共有サーフェス中心構成** | 単一プロセス・単一ビューではオーバーキル |

---

## 8. 採用の優先度（今回の結論）

| 区分 | 案 |
|------|-----|
| **今すぐ第一候補（次に試す）** | **DirectWrite（D3D11 と共有する D2D デバイスコンテキスト経由が定番）** — 日本語デバッグ表示との適合が最優先 |
| **次点** | **中間テクスチャ**で GDI をオフスクリーンに描き 1 枚貼る（移行の橋） |
| **今は採らない** | 共有サーフェス主役、SDF 全文エンジン、ASCII 限定スプライトのみ（要件とズレ） |

---

## 9. 次段 T33 に渡す最小結論

- **描画順の目標**: `clear` → **D3D 上で overlay** → **Present 1 回**（GDI の事後重ねをやめる）。
- **resize**: `WindowsRenderer_Resize` に合わせて **DWrite/D2D の再バインド／再レイアウト**を同じライフサイクルに載せる。
- **第一実験**: 「1 行だけ DirectWrite を RTV に描く」プロトタイプでデバイス共有とリサイズを検証し、長文・スクロールはその後。
