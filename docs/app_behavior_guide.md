# アプリ挙動ガイド（InputPlatformLab）

この文書は、**設計方針のテンプレートではなく**、現行の MainApp を起動したときに「何が見え、どの操作で何が起き、表示やログが何を指すか」を把握するための説明書です。詳細な API 一覧は [api_reference.md](api_reference.md)、ディレクトリの意味は [architecture.md](architecture.md) を参照してください。

---

## このアプリの目的

InputPlatformLab は、**製品ゲームの UI ではなく**、入力・表示・コントローラ周りの**検証・観察・実験**を行うための Windows デスクトップ試作アプリです。

主に次のような対象を、**画面のデバッグ文字列**と **OutputDebugString 系のログ**で追いやすくしています。

- キーボード（Raw Input）とレイアウト表示用ラベル
- ゲームパッドを **論理入力**（仮想ボタン・スティック方向）に正規化したうえでの、メニュー試作（2×2）やスクロール操作
- モニタ・表示モード一覧、希望解像度との最近傍、ウィンドウスタイル／全画面表示などの切り替え試作
- D3D11 によるクリア色の表示（GDI テキストと合成）
- 接続しているコントローラの **VID/PID・推定ファミリー・パーサ種別** などの識別表示

挙動は実装の進捗に合わせて変わり得るため、本書は「現状の見え方」に基づき、**断定できない部分は書かない**ことにします。

## 画面の見方

メインクライアント領域は、上から概ね次のブロックで構成されます（スクロールで下に続く）。

- **サンプルメニュー状態**  
  `menuOpen` / `selection` / `last event` / 2×2 の `@` 表示など。仮想入力のメニュー試作の状態をそのまま表示します。
- **T14 Displays**  
  モニタ数・選択モニタ・モード一覧（表示は「行数制限」があり、スクロールで全体を見ます）。
- **T15 nearest resolution**  
  希望解像度プリセットに対する、一覧上の最近傍モード。差分や一致の有無が分かります。
- **T16 window** 周辺の説明  
  ウィンドウ作成時のサイズ・DPI・再作成など、ログと対応する画面説明。
- **T17 presentation**  
  プレゼンテーション（Windowed / Borderless / Fullscreen など）の**候補・適用結果**の表示。キーボードの F6 / Enter との連携説明が並ぶことがあります。
- **T18 controller identify**  
  Raw Input で見つかった HID ゲームパッドと XInput の先頭スロットなど、**1 台分の識別情報**。
- **スクロール用オーバーレイ**（下端付近）  
  `[scroll]` で始まるブロック。コンテンツ高・スクロール位置・F7/F8 のジャンプ目安などが並びます。

背景は D3D のクリア色、テキストは GDI で **透明背景**に近い描画です。全画面を白で塗りつぶすことはしません。

## 操作一覧（キーボード / コントローラー）

### キーボード（Raw Input 経由で論理フレームに反映されるもの）

タイマー境界で「前フレームと今フレーム」のエッジが取られるキーが中心です（詳細はコード上の `KeyboardActionState`）。

| 操作 | 主な結果（メニュー閉・開で異なる部分は後述） |
|------|-----------------------------------------------|
| **Tab** | メニュー試作の `menuOpen` をトグル。 |
| **Enter** | メニュー開いているときは「確定」イベント。閉じているときは T17 の適用を **ラッチ**し、タイマーで消費される経路があります。 |
| **Backspace** | メニュー試作の「キャンセル」相当（メニューを閉じる等）。 |
| **矢印（↑↓←→）** | メニュー**開**き: 2×2 の選択移動。メニュー**閉**き: ↑↓ は T14 のモード行スクロール、←→ は T15 の希望プリセットの切り替え。 |
| **PgUp / PgDn** | 縦スクロール（ページ単位）。 |
| **Home / End** | スクロール先頭 / 末尾。 |
| **F5** | T17 のモード切替や適用には**使わない**旨のリマインダ用（ログ・画面表示）。 |
| **F6** | メニューが閉じているとき、T17 のプレゼンテーション**候補を循環**。 |
| **F7** | T17 ブロック付近が画面上部に来るようスクロール（上余白あり）。 |
| **F8** | T17 ブロックが画面中央付近に来るようスクロール。 |
| **マウスホイール** | 縦スクロール。 |

### コントローラー（XInput 先頭接続スロット + 必要に応じて HID）

論理入力ポリシーでは、**Confirm = South**（押下エッジ）、**Cancel = East**、**Menu = Start**、**移動 = DPad 優先、なければ左スティックの 4 方向**です。

キーボード由来のフレームとパッド由来のフレームは **マージ**されます。移動量は「パッド側が非ゼロならパッド優先」、確定・キャンセル・メニューは **キーとパッドのどちらかが押されれば発火**します。

## 操作すると何が起きるか

### メニュー（`menuOpen`）の開閉

- **Tab** で `menuOpen` が切り替わります。
- **開いているとき**、矢印は **2×2 グリッドの選択**だけに使われ、T14 のモード行スクロールや T15 のプリセット変更は **行いません**（コード上、メニュー閉時専用の分岐）。
- **閉じているとき**、↑↓ は T14 の可視行まわりのスクロール、←→ は T15 の希望解像度プリセット変更に使われます。

### T17（プレゼンテーション）

- メニューが**閉じている**ときだけ、**F6** でプレゼンテーション（表示モード）の**候補が循環**します。
- **Enter** の押下は、メニューが閉じているときに T17 の「適用」を **ラッチ**し、タイマー処理内で一度だけ適用される流れがあります（ログに APPLY 系が出ます）。
- **F5** は T17 の循環・適用には使いません（リマインダのみ）。

### スクロール

- **PgUp / PgDn / Home / End**、**マウスホイール**は、縦スクロールバーに連動して本文が動きます。
- **F7 / F8** は、T17 ブロックが画面の上付近／中央付近に見えるようスクロール位置を調整します。

## 画面内の各表示の意味

| 表示 | 何を表すか | どう変化するか | 何を見るためか |
|------|------------|----------------|----------------|
| **menuOpen** | メニュー試作が開いているか | Tab で 0/1 が切り替わる | メニューと T14/T15 の操作の優先が切り替わるか |
| **selection** | 2×2 の選択セル | メニュー開時のみ矢印で変化 | 仮想入力の移動が効いているか |
| **last event** | 直近のメニュー試作イベント | メニュー操作で更新 | どのイベントが起きたか |
| **2×2 selection** | 4 セルのどこに `@` があるか | 同上 | 同上 |
| **T14 Displays** | モニタとモード一覧 | 選択行・先頭行・スクロールで見える範囲が変わる | 解像度一覧と選択インデックス |
| **T15 nearest resolution** | 希望に対する最近傍 | プリセット変更で更新 | 希望と実モードの対応 |
| **T16 window** 周辺 | ウィンドウ寸法・DPI・再作成 | モード適用や再作成で変わる | 実寸と論理クライアントの関係 |
| **T17 presentation** | 表示モード・適用・キー案内 | F6 / Enter / 適用結果で変わる | どのモードが候補・適用されたか |
| **T18 controller identify** | 1 台の HID + XInput 情報 | デバイス接続変化で変わる | 識別・パーサ・サポート表示 |
| **scroll ブロック** | コンテンツ高・スクロール位置・F7/F8 目安 | リサイズ・スクロールで変わる | 長文がどこまでスクロールできるか |

## デバッグ出力・コンソール出力の見方

Visual Studio の出力ウィンドウ、またはデバッガーに付随するログに、次のような接頭辞が現れます。

| 接頭辞 | どの層か | 正常時の目安 | 注目するタイミング |
|--------|------------------------|----------------|--------------------|
| **[D3D11]** | D3D11 初期化・リサイズ・Present | init / resize / present が失敗しないログ | 初期化失敗・真っ黒画面 |
| **[SCROLL]** | スクロール情報・スクロールバー | リサイズや WM_PAINT 後にクライアント高・コンテンツ高が整合 | スクロールがおかしいとき |
| **[T14VIEW]** | T14 の自動スクロール追従 | レイアウト未準備なら skip | 選択行の追従を確認するとき |
| **[T16]** | ウィンドウ作成・サイズ・再作成 | CreateWindow 前後のログが期待値に近い | 解像度・DPI 不整合 |
| **[T17]** | プレゼンテーション循環・適用 | 意図したモード名・適用シーケンス | 全画面・ボーダーレス等の切替 |
| **[T18]** | コントローラ識別の変化 | 接続時に 1 行 | 接続デバイスが変わったとき |
| **[PS4Bridge]** 等 | DS4 HID から仮想入力への橋渡し調査 | 接続・レポート内容に応じて変化 | DS4 挙動を追うとき |
| **[HIDgen]** | 汎用 HID（検証度が tentative） | スロットルされた要約行 | 未知パッドの存在確認 |
| **VirtualInput…** | 論理入力・メニュー試作のログ | 状態遷移が操作に一致 | 入力統合の確認 |
| **PhysicalKey:** | 物理キー 1 イベントのラベル | キー押下時に行が出る | キーボードレイアウト表示用 |

## コントローラー種別表示の意味

| 語 | このアプリでの意味 |
|----|---------------------|
| **Xbox / PS4 / PS5 / Nintendo / XInputCompatible / Unknown** | `GameControllerKind` に相当する**推定ファミリー**。表示名やラベル表の切り替えに使います。 |
| **parser** | 入力経路の種類（例: XInput、既知 DS4 HID、汎用 HID）。コード上 `ControllerParserKind`。 |
| **support** | 検証済みか暫定か。**verified** は実機・マップまたは API 契約で固定したもの、**tentative** は受け皿・未固定の可能性がある旨の意味合いです。 |
| **Ds4KnownHid** | 特定 VID/PID で **DS4 形式 HID レポート**として解釈する経路。 |
| **GenericHid** | 上記以外のゲームパッド HID。ビットマップは固定しない前提のログ寄り。 |

### 「PS4」表記について

- **PC 上で DS4 を USB/Bluetooth として Raw Input / HID で扱う**文脈（VID 054C 等）と、**PlayStation 4 本体向けの実行ファイル**という意味は別です。本アプリの画面・ログは前者が主です。
- コンソール向けプラットフォーム移植の話は、このリポジトリの範囲を超える場合があります。

## 新しいコントローラ実装を追加する時の変更箇所

ここは「**必ずここを書き換えよ**」という確定仕様ではなく、**未検証のデバイスを足すときに最初に見る候補**を整理したものです。実際の作業はデバイス・プロトコルに依存します。

### ありうる変更の種類

- 既存ファミリー表記のまま **VID/PID 行を追加**する（テーブル拡張）
- **新しいファミリー**や **parser 種別**を増やす（型・分岐・ラベルが増える）
- **Raw レポートのビット解釈**を追加・修正する（DS4 以外の HID など）
- **VirtualInputSnapshot への変換**を追加する（新しいパス）
- **ボタン表示ラベル**（ファミリー別）の追加
- **T18 や [HIDgen] ログ**の見え方調整

### 調査・改修の入口（責務ベース。現行の置き場所は一例）

フォルダ構成の考え方は [architecture.md](architecture.md) を前提に、**どの責務に手を入れるか**を先に決め、そのあと **現行リポジトリではどのファイルに相当するか**を辿ると迷いにくいです。下表の「入手先」は、**現在の MainApp プロジェクト**（`app/InputPlatformLab/MainApp/`）から見た相対パスの例であり、将来ディレクトリを動かした場合は **同じ責務が別パスに移る**可能性があります。

| 関心・責務 | 入手先の例（現行） | よくある調整 |
|------------|-------------------|--------------|
| VID/PID と parser / support の対応、ファミリー推定 | `ControllerClassification`（`include/…/ControllerClassification.h` と `src/ControllerClassification.cpp`） | テーブル行、解決関数、分類ロジック |
| 論理ボタン列挙やファミリー名の拡張 | `GamepadTypes.h`（`include/`） | `GameControllerKind` 等の追加（影響範囲は広くなりがち） |
| 論理入力ポリシー・スナップショット・キーとパッドのマージ | `VirtualInputNeutral`（`include/VirtualInputNeutral.h` と `src/VirtualInputNeutral.cpp`） | ポリシー、フレーム組み立て |
| Raw Input / XInput / 既知 HID の取り込み、仮想入力・識別・ログ文字列の組み立て | メイン実装（現行は `src/MainApp.cpp` に集約されている部分が大きい） | WM_INPUT・タイマー経路、デバイス別の橋渡し |
| 中立ヘッダの **参照の入口**（どの型がどこから来るかの把握） | `InputCore.h`（`include/`） | **ロジック本体の追記先というより**、include 関係の地図として見ることが多い |
| 描画・GDI オーバーレイ（入力パースそのものではない） | `WindowsRenderer` / `Win32DebugOverlay`（`include/platform/win/` と `src/platform/win/`） | 画面に何を出すか・ログと並べて読みやすくするか |

**PC 上の新しい HID ゲームパッド**を扱う追加と、**別 OS やゲーム機本体向けの platform 追加**は別問題です。後者は [architecture.md](architecture.md) の「プラットフォーム」節を参照してください。

## 読むときの注意（PS4 表記など）

- 画面・ログの **PS4 / DS4** は、**Windows 上でコントローラを HID として読む**話と、**本体向けビルド**の話を混同しないでください。
- **verified / tentative** は「実装の完成度」ではなく、**テーブルと検証の置き方**の区別に近いです。

## この文書の利用場面

- 初めて起動したとき、**画面のブロックの意味**を掴む
- 入力・表示切替の**操作確認**をする
- **ログの読み方**の前提をそろえる
- 他タイトルに流用する前に、**現行 Windows 版の挙動**を把握する
- プラットフォーム追加や大規模改修の前に、**Windows 版の観察**の基準にする
- **新しいコントローラ対応を追加する前**に、どのファイルを調査の入口にしやすいかを決める
